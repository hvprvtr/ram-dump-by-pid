# ram-pid-reader

Чтение памяти процесса **по PID/имени** через драйвер дампера физпамяти
(WinPmem), **без `OpenProcess`/`PROCESS_VM_READ`** в коде инструмента.

Под капотом — `memprocfs` + `LeechCore`: LeechCore поднимает подписанный
драйвер WinPmem (device-строка `PMEM://<путь к winpmem_x64.sys>`), а MemProcFS
сам делает `PID → CR3 → page-table walk` поверх физических чтений. Мы лишь
дёргаем `read(va, size)`.

## Архитектура

| Слой | Файл | Зависит от Windows? |
|------|------|---------------------|
| Чистая логика (выбор процесса, валидация, hexdump, обход регионов для дампа) | `rampidreader/core.py` | нет — тестируется на моках |
| Адаптер memprocfs/LeechCore | `rampidreader/backend.py` | да (ленивый импорт) |
| CLI | `ramreader-by-pid.py` | да |

## Установка

На целевой Win11 (от администратора):

```powershell
pip install -r requirements.txt
```

## Драйвер winpmem

pip-пакет `memprocfs` **не содержит** драйвер winpmem. Его нужно извлечь из
`winpmem_mini` (флаг `-d`) и передать инструменту через `--driver`:

```powershell
winpmem_mini_x64.exe -d C:\path\winpmem_x64.sys
```

LeechCore собирает из этого device-строку `PMEM://C:\path\winpmem_x64.sys`.

## Использование

```powershell
# hex-дамп 256 байт по виртуальному адресу
python ramreader-by-pid.py --name notepad.exe --addr 0x7ff600000000 --size 256 --driver C:\path\winpmem_x64.sys

# от базы модуля + смещение, сырыми байтами в файл
python ramreader-by-pid.py --pid 1234 --module ntdll.dll --offset 0x1000 --size 4096 --out dump.bin --driver C:\path\winpmem_x64.sys

# карта регионов адресного пространства (VAD)
python ramreader-by-pid.py --pid 1234 --vads --driver C:\path\winpmem_x64.sys

# ПОЛНЫЙ дамп процесса — нужны только --pid/--name и --out
python ramreader-by-pid.py --pid 1234 --dump-all --out proc.bin --driver C:\path\winpmem_x64.sys

# продвинуто: сырая device-строка LeechCore (FPGA, дамп-файл и т.п.)
python ramreader-by-pid.py --pid 1234 --vads --device FPGA
```

### Полный дамп процесса (`--dump-all`)

Обходит все регионы (VAD) процесса и пишет читаемую память в `--out`. Ничего
предварительно вычислять не нужно — только PID (или имя) и файл. Рядом
создаётся карта `<out>.map`, связывающая виртуальные адреса с файлом:

```
# va length file_offset
0x00007ff7019c0000 65536 0
0x00007ff7019d0000 4096 65536
...
```

Перед чтением каждого региона берётся «проба» первой страницы: если она
недоступна, регион (как правило, reserved/незакоммиченная память — бывает по
несколько гигабайт) пропускается целиком, без перебора чанками. Это защищает от
зависания на дампе.

Сам регион читается чанками. Бэкенд обрывает чтение на первой невыделенной
странице и возвращает меньше запрошенного, поэтому курсор сдвигается на число
**реально прочитанных** байт, а упёршись в дыру — на одну страницу вперёд. Так
committed-данные за дырами не теряются (важно: иначе из крупных фрагментированных
регионов сохранялось бы только начало). Дыры в `.bin` не попадают — файл содержит
только реально присутствовавшую память, а карта `.map` хранит точные адреса
каждого блока. В конце печатается сводка: сколько регионов снято целиком,
частично и пропущено.

Коды возврата: `0` — успех, `1` — ошибка чтения/процесса, `2` — ошибка аргументов.

> Проверено на Win11 build 26200 (winpmem 2.0.1): побайтовое чтение модулей
> совпадает с эталонным дампом ОС (procdump), а полный дамп `--dump-all` снят
> с нескольких процессов (notepad, calc, powershell).

## Тесты

Идут на любой ОС, драйвер не нужен (всё на моках), включая злые тесты:

```bash
python3 -m venv .venv && .venv/bin/pip install -r requirements-dev.txt
.venv/bin/python -m pytest -q
```

## Требования к окружению

* Запуск от администратора (загрузка драйвера).
* Драйвер WinPmem должен загружаться (на Win11 с HVCI это главный фильтр).
* `--driver <winpmem_x64.sys>` собирает device-строку `PMEM://<path>`, по которой
  LeechCore поднимает свою копию драйвера WinPmem (голый `pmem` не подходит).
